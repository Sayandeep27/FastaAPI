# FastAPI HTTP Errors Guide

A quick reference guide to the most commonly used **HTTP status codes in FastAPI APIs**. This document explains what each error means and provides a practical real‑world example.

---

## Why HTTP Status Codes Matter

HTTP status codes help the **client (frontend, mobile app, or another service)** understand what happened after making a request to the API.

They are essential for:

* Debugging API issues
* Proper error handling
* Building reliable backend services
* Writing clean REST APIs

FastAPI uses standard HTTP status codes along with **automatic validation using Pydantic**.

---

## Common FastAPI Error Codes

| Code | Meaning          | Short Explanation                                                              | Real Example                                                                               |
| ---- | ---------------- | ------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------ |
| 400  | Bad Request      | The request sent by the client is incorrect or malformed.                      | Sending invalid query parameters like `/users?page=abc` where `page` should be an integer. |
| 401  | Unauthorized     | The user is not authenticated. A login or valid token is required.             | Accessing `/profile` without providing a JWT token.                                        |
| 403  | Forbidden        | The user is authenticated but does not have permission to access the resource. | A normal user trying to access an admin‑only API like `/admin/users`.                      |
| 404  | Not Found        | The requested resource does not exist.                                         | Requesting `/users/100` when user with ID `100` does not exist in the database.            |
| 409  | Conflict         | The request conflicts with existing data in the system.                        | Trying to register with an email that already exists.                                      |
| 422  | Validation Error | FastAPI automatically throws this when request data fails Pydantic validation. | Sending `"age": "twenty"` when the model expects `age: int`.                               |
| 500  | Server Error     | An unexpected error occurred in the backend code.                              | A bug like `10/0` or database connection failure causing the server to crash.              |

## How FastAPI Returns These Errors

FastAPI allows returning errors using **HTTPException**.

### Example

```python
from fastapi import FastAPI, HTTPException

app = FastAPI()

@app.get("/users/{user_id}")
def get_user(user_id: int):

    if user_id != 1:
        raise HTTPException(
            status_code=404,
            detail="User not found"
        )

    return {"id": 1, "name": "John"}
```

### Response

```json
{
  "detail": "User not found"
}
```

## Example: Custom 400 Error

```python
from fastapi import HTTPException

if age < 18:
    raise HTTPException(
        status_code=400,
        detail="User must be at least 18 years old"
    )
```

## Example: Authorization Error (403)

```python
if user.role != "admin":
    raise HTTPException(
        status_code=403,
        detail="Admin access required"
    )
```

---

## Example: Conflict Error (409)

```python
existing_user = db.find_user(email)

if existing_user:
    raise HTTPException(
        status_code=409,
        detail="Email already registered"
    )
```

## FastAPI Automatic Validation Error (422)

FastAPI automatically returns **422** when request data does not match the Pydantic model.

### Model

```python
from pydantic import BaseModel

class User(BaseModel):
    name: str
    age: int
```

### Invalid Request

```json
{
  "name": "Alex",
  "age": "twenty"
}
```

### Response

```json
{
  "detail": [
    {
      "loc": ["body", "age"],
      "msg": "value is not a valid integer",
      "type": "type_error.integer"
    }
  ]
}
```

## Best Practices

1. Use **400** for client mistakes.
2. Use **401** when authentication is missing.
3. Use **403** when authentication exists but permission is denied.
4. Use **404** when a resource does not exist.
5. Use **409** for duplicate or conflicting data.
6. Let **FastAPI automatically handle 422 validation errors**.
7. Use **500** only for unexpected server failures.
